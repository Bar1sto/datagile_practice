# CVE Vulnerability Tracker — Analysis

# Цель
    Сервис ежедневно синхронизирует CVE из NVD, хранит их в PostrgeSQL, и дает API для поиска, просмотра и статистики

## нужные ссылки:

    NVD (NIST): https://nvd.nist.gov/developers/vulnerabilities 

    OSV: https://osv.dev/

    ФСТЭК России (БДУ): https://bdu.fstec.ru/export 

# Синхронизация данных
### Must Have 
#### СД-01
    при вервом запуске сервис выполняет начальную загрузку CVE-записей из NVD API за последние 12 месяцев
#### СД-02
    сервис автоматически синхронизирует данные по расписанию (интервал задаётся в конфигурации) добавляет новые записи, обновляет изменившиеся

#### СД-03
    каждая синхронизация фиксируется в БД: дата и время запуска, количество добавленных и обновленных записей, статус завершения, описание ошибки при наличии
#### СД-04
    повторный запуск синхронизации не приводит к дублированию данных

## Should Have
#### СД-05
    сервис поддерживает загрузку и переодическую синхронизацию данных из дополнительных источников: OSV (osv.dev) и БДУ ФСТЭК (bdu.fstec.ru). каждый источник имеет собственную структуру данных и формат идентификаторов. разработчику необходимо самостоятельно определить схему хранения, стратегию нормализации и подход к связыванию записей из разных источников по уязвимости

# REST API
### Must Have
#### АП-01
    GET /cve/{cve_id} - возвращает полную запись об уязвимости по идентификатору. при отсутствии записи возвращает 404
#### АП-02
    GET /cve - возвращает постраничный список записей. поддерживаемые фильтры - вендор, продукт, уровень критичности (LOW, MEDIUM, HIGH, CRITICAL), диапозон дат публикации. структуру параметров пагинации определить самостоятельно
#### АП-03
    GET /stats - возвращает агрегированную статистику по накопленным данным. состав метрик определить самостоятельно исходя из доступных данных
#### АП-04
    все ендпоинты возвращают унифицированную структуру ошибок
#### АП-05
    API документируется автоматически. swagger UI доступен без дополнительной настройки

# нефункциональные требования
### Must Have
#### НФ-01
    все настройки (параметры БД, интервал синхронизации и пр.) задаются исключительно через конфигурационный файл
#### НФ-02
    полный стек сервиса запускается командой docker-compose up без предварительных ручных действий
#### НФ-03
    покрытие тестами ≥ 60%. тесты охватывают бизнес-логику и API-эндпоинты
#### НФ-04
    репозиторий содержит README с описанием архитектуры решения, инструкцией по запуску и примерами запросов к API
### Should Have
#### НФ-05
    для каждого запуска синхронизации фиксируется время выполнения и пиковое потребление памяти. показатели сохраняются в бд вместе с логом синхронизации (СД-03) и доступом через GET /stats

## Этапы и контрольные точки
### 1 неделя
    развертывание и аналитика - репозиторий настроен. схема бд спроектированна, миграции применяются. данные из NVD загружаются вручную
### 2 неделя
    API и синхронизация - все эндпоинты реализованы и доступны через swagger. автоматическая синхронизация работает по расписанию. логи синхронизаций пишутся в БД
### 3 неделя
    тесты и frontend - тесты написаны, покрытие ≥ 60%, линтер без ошибок, базовый фронт работает (ФЕ-01 и ФЕ-02)
### 4 неделя
    финализация - полный стек поднимается через docker-compose. README.md готов, при наличии времени - Should Have задачи. Финальное код ревью
